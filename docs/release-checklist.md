# Production readiness checklist

Every item below is a **release blocker** until checked with evidence (CI run ID, runbook sign-off, security review artifact, or staging test report). Developer validation mode (local JSON store, header auth) does **not** satisfy these gates.

**Do not mark a `PROGRESS.md` task `[x]`** when the row for that area remains unchecked here.

The enterprise gap backlog in [`product/08-enterprise-production-gap-backlog.md`](product/08-enterprise-production-gap-backlog.md) is also release-governing. P0 items there must be implemented or explicitly deferred with owner and sign-off before a production readiness claim.

**Readiness signals (do not conflate):** npm validators prove structural metadata shape only. Staging attestation `production_ready: true` means the profile’s required release-evidence kinds are accepted in inventory — **not** production promotion. Gap audit `production_ready: true` additionally requires a **submittable** accepted inventory plus closed documented checklist/release-plan rows (gate closeouts are evidence-derived from `output/release-evidence/records.json`, not blind doc rewrites). Dry-run collector output (`collect-release-evidence.mjs --dry-run`) is contract-shaped preview only and cannot satisfy attestation or gap audit. Local production-like readiness uses internal SOC metadata evidence (`npm run staging:local:attest`) and hosted customer-production-like readiness uses Railway Postgres + bundled staging OIDC (`npm run staging:hosted:attest`) — not external provider signoff. Per-customer enterprise IdP mapping, live notification provider credentials, hosted KMS/HSM, and customer cloud/CDN integrations remain **tenant onboarding** — separate from repo checklist rows marked **Closed (staging execution)** here.
