# Security posture: developer validation vs production gates

Summary of controls in **developer validation mode** and **production release blockers** that must be closed before GA. See [`docs/security/01-security-model.md`](01-security-model.md) for the target production model.

## Production release blockers

| Gap | Risk | Acceptance criteria | Owner |
|---|---|---|---|
| Developer validation headers (`x-tenant-id`, `x-role`) | Spoofing tenant/role | `oidc-jwt` only in prod (startup refuses headers); **remaining:** header-only negative test in prod-like deployment with audit signoff | Backend + Security |
| JSON file store (developer runtime) | No RLS, race conditions | PostgreSQL runtime boundary now starts and injects migrated services in `postgres` mode; guarded JSON-backed route families must be migrated before release; migrations applied in staging/prod + tenant isolation under load remain required | Backend + DB |
| Envelope key management (production) | Env-key AES-256-GCM envelopes implemented for integration secrets (`ASTRANULL_SECRET_ENCRYPTION_KEY`); bootstrap/`svc_` remain salted hashes with dev-store rotate API; decrypt-for-use internal only | KMS/HSM-backed keys, envelope automated rotation policy, Postgres durability, key-rotation drill | Security |
| API rate limits (production edge) | Abuse at scale; limiter bypass at edge | Gateway/WAF + service limiter validated under load; `ASTRANULL_TRUST_PROXY_HEADERS=1` only behind trusted proxy | Backend + Security |
| Agent package signing / update trust (partial) | Supply chain; metadata and host primitives do not replace CDN custody ops or unattended fleet rollout | `scripts/package-agent.mjs` builds generic `.tar.gz`, manifest, optional Ed25519 `.manifest.json.sig` (`AG-005`); `POST/GET /v1/agent-update-trust-keys`, revoke, active **tenant trust-key** gate (`untrusted_signing_key`), release `distribution` + poll `download` URLs, shared signed-manifest verification, and Postgres route persistence through `runtime.services.agentUpdates`; agent `--verify-update-manifest`, `--apply-update-manifest`, and `--download-and-apply-update` (HTTPS download hardening, `current.json`) (`AG-014`). **Remaining:** production installer/update-daemon trust-key enforcement, hosted artifact custody/CDN/mirror ops runbooks and staging evidence, unattended daemon restart and fleet rollout drills, deb/rpm/container signing, key ceremony/custody/rotation/revocation drill, distro matrix (`AG-002`) | Agent |
| In-process probe simulation (default dev/CI path) | Not fleet-scale safe testing; signed bounded profiles exist but fleet E2E open | Signed-worker fleet in staging/prod; simulation not sole production path | Detection + Backend |
| High-scale dry-run adapter | Local dry-run cannot satisfy production execution | `src/contracts/governedExecutionAdapter.mjs` validates production adapter registration metadata and required capabilities; remaining: connected partner/internal adapter, staging start/stop/kill-switch proof, telemetry/evidence export proof | SOC |
| UI role dropdown | No live enterprise IdP login evidence | IdP integration, conditional access, and staging signoff | UX + Backend |

## Edge protection release evidence (metadata-only)

Use `node scripts/edge-protection-evidence.mjs` before promoting builds that depend on customer-facing edge controls. The CLI is local-only: it does not call cloud providers and does not read credentials from the environment.

**Acceptable metadata**

- Release identifiers and ISO timestamps (`release_id`, `signoff_at`, per-control `validated_at`).
- Provider-neutral summaries: `edge_stack_summary` (WAF, API gateway, CDN, or equivalent), `rate_limiting_summary`, `logging_redaction_summary`, and the per-control fields defined in `src/contracts/edgeProtectionBaseline.mjs` (for example `tls_policy`, `rule_family_summary`, `log_destination`, `owner`).
- Opaque evidence references (`evidence_uri`, `siem://…`, `evidence://…`) that point reviewers to out-of-band artifacts without embedding payload content.

**Forbidden raw data**

- HTTP headers or bodies (including nested `raw_headers`, `request_body`, `response_body`, or provider export dumps).
- Log lines, packet captures, or payloads (`raw_log`, `log_lines`, `packet_payload`, and keys matching `raw_*`).
- Credentials, tokens, cookies, authorization material, or secret-bearing fields (rejected by the baseline contract before write).
- Cloud API responses, live configuration snapshots, or any field that would require provider credentials to collect inside this CLI.

Passing CLI validation only proves the evidence *shape* is safe for release review. Production still requires deployed edge configuration, staging abuse/load results, log routing proof, and explicit security signoff.

## Operator guidance

- Do not expose developer validation HTTP to the public internet.
- Treat bootstrap tokens, `svc_` service-account tokens, and agent credentials (`agc_v1.…` or legacy `agc_…`) as secrets; rotate after validation sessions; revoke automation tokens via `POST /v1/service-accounts/:id/revoke` when no longer needed.
- Run `make verify` before promoting builds; review audit log for unexpected `rbac.denied` spikes.
