# Release Plan

Releases are **production readiness milestones**, not demo drops. Each milestone requires [`docs/release-checklist.md`](../release-checklist.md) sign-off, honest [`PROGRESS.md`](../../PROGRESS.md) status, and disposition of P0 items in the enterprise gap backlog ([`08-enterprise-production-gap-backlog.md`](08-enterprise-production-gap-backlog.md)).

## Release 1: Core platform (production minimum)

### Goal

A customer can authenticate with enterprise SSO, install a signed agent, declare target groups, run safe validation checks via governed probes, and receive evidence-backed findings stored in PostgreSQL.

### Must ship (release blockers if missing)

| Area | Required features |
|---|---|
| UI | Onboarding, target groups, agent install, dashboard, test runs, findings. |
| Backend | Tenant, SSO auth, tokens, agent registry, target group APIs, test planner, correlation, findings, Postgres persistence. |
| Agent | Linux host install (signed), registration, heartbeat, canary/packet metadata observation. |
| Detection | Direct-origin bypass, protected-path canary, forbidden port, WAF marker, basic L7 checks with external probe workers, per-vector safety policy, and verdict explanations with placement confidence. |
| Reporting | Run summary report and finding export with redaction. |
| Security | RBAC, audit logs, encrypted secrets, safe-rate caps, tenant isolation tests. |

## Release 2: Enterprise operations

### Goal

A multi-team customer can operate AstraNull continuously with integrations and retention.

| Area | Required features |
|---|---|
| UX | Full pages/tabs, visualization suite, reports, notifications. |
| Backend | Queues, scheduling, report generator, audit evidence retention, DR, real notification providers, usage/accounting foundations, and admin operations controls. |
| Agent | Docker, Helm, package repos, upgrades, placement diagnostics, and agent confidence scoring. |
| Detection | DNS, L3/L4, API, TLS/connection, HTTP/2/HTTP/3 readiness per the expanded vector backlog. |
| SOC | High-scale request intake, SOC-009 authorization proof workflow, and approval gates (governed execution still SOC-only). |

## Release 3: SOC-gated high scale (GA for high-scale program)

### Goal

AstraNull safely manages approved high-scale readiness assessments with immutable authorization archives.

| Area | Required features |
|---|---|
| SOC | Approval state machine, runbook, kill switch, provider approval tracking. |
| Execution | Partner/internal **governed** adapter interface with constraints — no raw attack recipes. |
| Evidence | Live telemetry, post-test reports, immutable authorization archive, evidence chain-of-custody, and signed report hashes. |
| Compliance | Audit, retention, customer authorization templates, legal review checklist, SOC 2/ISO/DORA/NIS2 report mappings. |

## Open production release gates (all releases)

These gates stay **open** until externally evidenced and signed off. Local scaffolding, developer-validation tests, `npm run release:sample-evidence` rehearsal fixtures, and accepted metadata in a dev JSON store **do not** close a row.

**Three readiness signals (do not conflate):**

| Signal | Meaning |
|---|---|
| Structural validation | npm evidence validators and bundle/attestation CLIs accept metadata shape and forbidden-field rules only. |
| Profile inventory complete | `production_ready: true` on staging attestation (or `evidence_attestation_complete: true` on gap audit) means every kind required by the evaluated profile is present, contract-valid, and **accepted** in the tenant ledger — **not** production promotion. |
| External production promotion | Checklist rows closed, staging/security/SOC/legal/provider/KMS/signing execution, and release-manager signoff. Gap audit `production_ready: true` additionally requires documented checklist gates to be closed; neither attestation nor gap audit alone substitutes for this. |

| Gate | Owner | Evidence / artifact | Status |
|---|---|---|---|
| Product and API contract accuracy | Product + Backend | Published docs match production auth, persistence, and release-evidence contracts; change record in release notes | **Open** — ongoing doc alignment |
| P0 enterprise gap backlog | Product + Engineering lead | Each P0 in [`08-enterprise-production-gap-backlog.md`](08-enterprise-production-gap-backlog.md) implemented or deferred with named owner and signoff reference | **Open** |
| Signed agent packages and install matrix | Agent + Release | Staging execution of install matrix on supported distros; accepted `agent_install_matrix` / `agent_sbom_provenance` from operators (not sample generators) | **Open** |
| Database migrations | DBA / Platform | Migrations applied in staging/production with runner proof; accepted `migration_apply`; rollback/fix-forward drill evidence (`rollback_fixforward`) | **Open** |
| Rollback and kill-switch drills | SOC + Platform | Tabletop or live drills with operator signoff; accepted `kill_switch_drill`, `operator_runbook_exercise`, `rollback_fixforward` | **Open** |
| Independent security review | Security | Completed third-party or penetration review with remediation tracker; accepted `third_party_security_review` tied to real report URIs | **Open** |
| SOC high-scale governance | SOC | High-scale approval/kill-switch workflows exercised in staging; accepted `governed_adapter`, `provider_approval`, `authorization_custody` where in scope | **Open** |
| Staging QA / E2E matrix | QA | Staging E2E matrix execution; accepted `staging_e2e_matrix`, `ui_accessibility_matrix`, `placement_confidence_staging` | **Open** |
| Staging readiness attestation (profile-aware) | Release + QA | Operator-attested evidence bundle; CLI: `node scripts/staging-readiness-attestation.mjs --input <evidence.json> --profile full\|safe-validation-ga\|high-scale-ga [--release-id rel] [--out file]` or `npm run release:staging-attestation`; API mirror: `GET /v1/production-release-evidence/attestation` (profile `full`). `production_ready: true` in output means required kinds for the profile are present and contract-valid in the inventory — **not** production sign-off | **Open** |
| Production readiness gap audit | Release + Security + SOC | `node scripts/production-readiness-gap-audit.mjs [--evidence bundle.json] [--release-id rel] [--out file] [--validate-only]` or `npm run release:gap-audit`; reports `evidence_attestation_complete` (profile inventory) separately from `production_ready`, which is true only when inventory is complete **and** documented checklist/plan gates are closed — external staging/security/SOC/legal promotion still required | **Open** |
| KMS/vault, edge, and control-plane release | Security + Platform | Accepted `kms_vault_posture`, `edge_protection`, `control_plane_container_release`, `gateway_load_abuse`, `secret_rotation_drill` from staging/operator runs | **Open** |
| Compliance and legal signoff | Legal + Compliance | Accepted `compliance_legal_signoff` and custody artifacts; board/auditor signoff where required | **Open** |
| Support and observability readiness | Support + SRE | Staffed playbook linked from customer portal; accepted `support_readiness`, `observability_slo`, `notification_provider_config` with real provider drills | **Open** |

**Rule:** No row above may be read as **production complete** without external evidence (staging execution, independent review, SOC/legal/provider/KMS/signing signoff) attached through governed custody — not local JSON validation alone.
